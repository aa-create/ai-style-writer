## 一、MemFire Cloud 建表（10 分钟）

打开你的 MemFire Cloud 项目，进入 **SQL 编辑器**，粘贴以下 SQL 一次性执行：

```sql
-- 1. 全局词汇库（跨材料类型共享，每用户一条）
create table user_global_phrases (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users(id) on delete cascade not null unique,
  phrases jsonb not null default '{}',
  created_at timestamptz default now(),
  updated_at timestamptz default now()
);

-- 2. 类型专属规则（每用户每类型一条，存规则+类型专属词汇+小标题模板）
create table style_rules (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users(id) on delete cascade not null,
  material_type text default 'propaganda' not null,
  rules jsonb not null default '{}',
  type_phrases jsonb not null default '{}',
  subtitle_styles jsonb not null default '[]',
  stats jsonb not null default '{"learned_count": 0}',
  created_at timestamptz default now(),
  updated_at timestamptz default now(),
  unique(user_id, material_type)
);

-- 3. 已学范文（带 material_type 标记来源）
create table learned_articles (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users(id) on delete cascade not null,
  material_type text default 'propaganda' not null,
  preview text not null,
  analysis jsonb not null default '{}',
  created_at timestamptz default now()
);

-- 4. 生成记录（带 material_type）
create table generations (
  id uuid default gen_random_uuid() primary key,
  user_id uuid references auth.users(id) on delete cascade not null,
  material_type text default 'propaganda' not null,
  raw_input text,
  extracted_points jsonb,
  output text,
  style_used text,
  created_at timestamptz default now()
);

-- 5. RLS（每张表用户只能读写自己的数据）
alter table user_global_phrases enable row level security;
alter table style_rules enable row level security;
alter table learned_articles enable row level security;
alter table generations enable row level security;

create policy "own_data" on user_global_phrases for all using (auth.uid() = user_id);
create policy "own_data" on style_rules for all using (auth.uid() = user_id);
create policy "own_data" on learned_articles for all using (auth.uid() = user_id);
create policy "own_data" on generations for all using (auth.uid() = user_id);
```

执行完后，在 MemFire Cloud 后台 **认证 > 设置** 里确认邮箱登录已开启（默认已开）。

### Round 2：统一 AI 接口封装（约 10 分钟）

```

   - 封装 SSE 流式响应的辅助函数
   - 将 AI 返回的 stream 转成 NextResponse 的 SSE 格式

代码示例（参考这个结构）：
   const response = await fetch(`${baseUrl}/v1/chat/completions`, {
     method: 'POST',
     headers: {
       'Content-Type': 'application/json',
       'Authorization': `Bearer ${apiKey}`,
     },
     body: JSON.stringify({
       model: model,
       messages: messages,
       stream: options?.stream ?? false,
       temperature: options?.temperature ?? 0.7,
     }),
   });

这样后续换模型只需改 .env.local 里三个变量即可。

### Round 3：文件解析工具（约 10 分钟）

```
创建文件解析工具 src/lib/file-parser.ts，要求：

1. 导出一个 parseFile(file: File): Promise<string> 函数
2. 支持三种格式：
   - .txt：直接读取文本
   - .md：直接读取文本
   - .docx：用 mammoth 库解析提取纯文本
     pnpm add mammoth
   - .pdf：用 pdf-parse 库提取文本
     pnpm add pdf-parse

3. 前端调用：用户上传文件后，在浏览器端解析出纯文本
   - docx 和 pdf 通过 arrayBuffer 读取后解析
   - 解析失败时给出友好错误提示

4. 限制文件大小 5MB 以内

注意：mammoth 在浏览器端可以直接运行，pdf-parse 需要在服务端。
所以 pdf 走 API 路由解析，txt/md/docx 在前端解析。

如果 pdf-parse 在前端不好用，就创建一个 POST /api/parse-pdf 的路由来处理。
```

### Round 5：写稿页面 — 两步生成（约 30 分钟）

这是核心页面，指令较长：

```
实现写稿页面 src/app/app/write/page.tsx，这是一个两步生成流程：

【页面布局】
上半部分：输入区
下半部分：结果区

【输入区】
- 一个大的 textarea，placeholder 写："把你想写的内容随便说说……比如：上周搞了个反诈宣传活动，联合派出所，在菜市场门口摆摊发传单500多份……"
- textarea 下方一行：左边是上传附件按钮（支持 .txt .md .docx .pdf），右边是小标题风格三选一按钮组（字诀式/动宾式/聚焦式）
- 上传文件后，解析出文本内容追加到 textarea 内容后面（或者显示文件名标签，内部拼接）
- 一个大按钮：「提取要点」

【第一步：提取要点】
点击「提取要点」→ POST /api/extract
- 后端 prompt：你是信息提取助手。从用户提供的文本中提取宣传文稿的关键要点，返回 JSON：
  {
    "topic": "核心主题",
    "participants": "参与单位/人员",
    "actions": "具体举措（数组）",
    "results": "成效/数据",
    "missing": "缺失但建议补充的信息（数组）"
  }
- 不要 stream，等完整返回
- 前端展示为可编辑的要点卡片：
  - 每个字段显示为一行，带标签和内容
  - 内容部分是可编辑的 input/textarea
  - missing 部分用黄色提示框显示："⚠ AI 建议补充：xxx"
  - 用户可以直接在卡片里修改内容
  - 两个按钮：「返回修改原文」和「确认，开始生成 →」

【第二步：生成全文】

- 后端：
  1. 通过 style-service.ts 读取该用户的全局词汇（user_global_phrases）和类型规则（style_rules）
  2. 合并词汇：global_phrases + type_phrases → merged_phrases，去重
  3. 拼装 system prompt（写作规范 + 合并后词汇库 + 小标题风格 + 禁止事项）
  4. user message 用第一步确认后的结构化要点
  5. 调用 AI，streaming 模式
- 前端：
  - 流式显示生成结果，用等宽字体或衬线字体
  - 生成完毕后显示：「复制」按钮
  - 存入 generations 表（带 material_type='propaganda'）

【API 路由】
- POST /api/extract：提取要点（非 streaming）
- POST /api/generate：生成全文（streaming SSE）

【Prompt 模板】

提取要点 system prompt：
"""
你是一个信息提取助手。用户会输入一段关于工作活动的描述，可能比较口语化、零散。
请从中提取以下要点，返回纯 JSON（不要 markdown 代码块标记）：

  "topic": "核心活动主题（简练概括）",
  "participants": "参与的单位或人员",
  "actions": ["具体举措1", "具体举措2"],
  "results": "成效、数据、反馈",
  "missing": ["建议补充的信息1", "建议补充的信息2"]
}
如果某项信息文本中没有提到，对应字段填空字符串或空数组。
missing 里列出你认为写宣传稿需要但用户没有提供的信息。
"""

生成全文 system prompt：
"""
你是一位资深的体制内宣传文稿写作专家。

【写作规范】
- 标题：{rules.title_pattern}，不超过{rules.title_max_length}字
- 结构：导语段 + {rules.body_paragraphs}段正文
- 每段结构：{rules.paragraph_structure}
- 每段不超过{rules.max_chars_per_paragraph}字

【本次小标题风格：{选中的风格名}】

【词汇库（优先使用这些词，已合并全局+类型专属词汇）】

"""

生成全文 user message：
"""
请根据以下要点写一篇宣传文稿：

主题：{topic}
参与单位：{participants}
具体举措：{actions join、}
成效：{results}
"""

页面要好看，交互要流畅，每个步骤之间有清晰的视觉分隔和状态变化。
```

【API 路由】

- PATCH /api/style：更新词汇（增删）

## 五、API 接口一览

| 方法 | 路由 | 输入 | 输出 | 流式 |
|------|------|------|------|------|
| POST | /api/extract | `{ text }` | `{ topic, participants, actions, results, missing }` | 否 |
| POST | /api/generate | `{ points, subtitleStyle }` | SSE 文本流 | 是 |
| POST | /api/analyze | `{ text }` | 六维分析 JSON | 否 |
| POST | /api/learn | `{ articlePreview, analysis, materialType }` | `{ newPhraseCount }` | 否 |
| GET | /api/style | `?type=propaganda` | 合并后的完整风格库（global + type） | 否 |
| PATCH | /api/style | `{ action, phrase, scope? }` | 更新后的 phrases | 否 |
| POST | /api/parse-pdf | FormData(file) | `{ text }` | 否 |

**PATCH /api/style 的 scope 说明：**
- `"global"`（默认）：增删 user_global_phrases.phrases.自定义
- `"type"`：增删 style_rules.type_phrases.自定义（MVP 暂不用）

## 六、关键 Prompt 汇总

### Prompt 1：提取要点

```
System: 你是一个信息提取助手。用户会输入一段关于工作活动的描述，可能比较口语化和零散。
请提取宣传文稿所需的关键要点，返回纯 JSON（不要 markdown 代码块标记）：
{
  "topic": "核心活动主题（简练概括）",
  "participants": "参与的单位或人员",
  "actions": ["具体举措1", "具体举措2"],
  "results": "成效、数据、反馈",
  "missing": ["建议补充的信息1"]
}
某项信息文本中没有提到就填空字符串或空数组。
missing 里列出写宣传稿需要但用户没提供的关键信息。

User: {用户输入的原文}
```

```
System:

不捏造数据/人名/地名，不用第一人称，不口语化，不超出用户事实范围。

主题：{points.topic}

## 七、踩坑提醒

1. **DeepSeek 返回 JSON 时偶尔带代码块标记**
   后端解析时加一层清洗：`text.replace(/```json|```/g, '').trim()`

2. **MemFire Cloud RLS 导致写入失败**
   确保用户已登录，并且用服务端客户端（service role key）或者在 RLS 策略中允许 insert

3. **本地开发连 MemFire Cloud 慢**
   MemFire 服务器在国内，本地开发速度正常。如果偶尔超时，检查网络连接。

4. **mammoth 在 Next.js 服务端组件中的兼容性**
   如果报错，改为在 API 路由里解析，不在客户端组件里直接 import

5. **文件上传大小**
   本地开发无限制。正式部署后在 Nginx 配置 `client_max_body_size 10m`。

6. **生产环境 Nginx 代理超时**
   AI 生成可能需要较长时间，Nginx 的 `proxy_read_timeout` 设为 120s 以上。
